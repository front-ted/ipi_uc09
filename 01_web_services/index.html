Web services: histórico e conceito

Este material visa auxiliar você a compreender o que são APIs (application programming interface) e web services. Para começar o aprendizado, imagine o seguinte cenário: um aplicativo de entrega de comida. Quando você escolhe um restaurante, o sistema precisa consultar o cardápio, calcular o frete e realizar o pagamento. Esses processos envolvem a comunicação entre APIs e web services, que permitem ao aplicativo buscar informações em servidores externos e integrá-las em tempo real. Na área de programação, o uso de APIs e web services é muito discutido, e essas tecnologias tornaram-se fundamentais para o desenvolvimento da maioria dos sistemas modernos, especialmente em softwares web e mobile.

1990

Nos anos 1990, com o crescimento da internet, surgiu a necessidade de que sistemas diferentes pudessem se comunicar de forma eficiente. Durante esse período, começaram a surgir soluções proprietárias para integração de dados, mas ainda faltava um padrão universal que facilitasse essa comunicação.

2000

No ano 2000, o protocolo SOAP (simple object access protocol) foi introduzido como uma solução padronizada para a troca de dados entre sistemas. Utilizando XML (extensible markup language) para estruturar as mensagens, o SOAP tornou-se muito adotado em sistemas empresariais, atendendo à necessidade de padronização da época.

2004

A partir de 2004, a arquitetura REST (representational state transfer) começou a ganhar destaque em razão de sua leveza e simplicidade em comparação ao SOAP. Nesse contexto, o formato JSON (JavaScript object notation) emergiu como o padrão preferido para transferência de dados, substituindo gradualmente o XML em diversas aplicações.

Atualmente

Atualmente, APIs e web services estão presentes em praticamente todas as áreas, desde aplicativos móveis até plataformas de e-commerce. Tecnologias covmo REST e ferramentas como o Postman facilitaram significativamente o uso e o teste de APIs, tornando-os mais acessíveis para os desenvolvedores.

Nos próximos tópicos, esses conceitos serão explorados em maior profundidade, mostrando como o REST e o JSON transformaram a comunicação entre sistemas e o impacto deles no desenvolvimento moderno.

O que é uma API?

API, ou interface de programação de aplicações, é uma ferramenta que permite a comunicação entre dois sistemas diferentes. Basicamente, uma API é composta de um conjunto de rotinas e padrões que permitem que um sistema acesse funcionalidades ou informações de outro.

Para entender melhor o conceito de API, imagine-se em um restaurante. Você está sentado à mesa (Sistema A) e quer fazer um pedido à cozinha (Sistema B, o servidor). Para isso, você comunica seu pedido ao garçom, que leva a solicitação para a cozinha, especificando todos os detalhes do prato escolhido. Quando a comida está pronta, o garçom traz o pedido de volta à sua mesa. Se você decidir pedir mais bebidas, o mesmo processo ocorre: o garçom se encarrega de comunicar a nova solicitação à cozinha e traz a resposta (o pedido) de volta.

Nesse contexto, o garçom representa a API. Ele é responsável por toda a comunicação entre você (Sistema A) e a cozinha (Sistema B). O ponto importante é que você, sentado à mesa, não precisa saber como o garçom se comunica com a cozinha nem como a cozinha prepara os pedidos. Você só precisa saber como fazer o pedido e confiar que o garçom o entregará corretamente.

As APIs simplificam a integração entre diferentes sistemas, permitindo que eles colaborem sem a necessidade de expor detalhes internos de funcionamento. Essa abordagem facilita o desenvolvimento de aplicações, conectando diversas plataformas, como serviços de pagamento, mapas, redes sociais e muito mais.

Ilustração comparativa mostrando o funcionamento de um sistema web e sua analogia com um restaurante. Na parte esquerda, o diagrama descreve a interação entre o front-end, a API e o back-end. Uma mão segurando um smartphone representa o front-end, que envia uma requisição para a API. A API é mostrada como um símbolo de engrenagem, conectando o front-end ao back-end, representado por servidores. O back-end processa a solicitação e envia uma resposta de volta ao front-end, finalizando o ciclo. Na parte direita, a analogia com um restaurante está representada em três etapas: 1. O front-end é simbolizado por um casal sentado em uma mesa de jantar. Eles solicitam o menu e fazem um pedido de refeição. 2. O garçom, representado por um atendente com uma bandeja, leva o pedido do casal até a cozinha. 3. A cozinha, ilustrada por um chef segurando um prato, prepara a refeição e devolve o pedido ao garçom, que entrega a comida ao casal. Setas mostram o fluxo contínuo de comunicação entre as partes em ambos os cenários, destacando como cada elemento desempenha uma função específica para atender à solicitação inicial.

Pode-se dizer que as APIs são ferramentas projetadas para simplificar o desenvolvimento de programas e aplicações. Elas fazem isso ao abstrair muitos dos elementos complexos de um software, permitindo que os desenvolvedores foquem o que realmente precisam: como utilizar os serviços oferecidos pela API em seus próprios projetos. A grande vantagem é que o desenvolvedor não precisa entender como a API executa suas operações internas, apenas como interagir com ela de acordo com sua documentação. É por isso que a documentação da API é importante. Embora existam padrões gerais, cada API tem as próprias regras e os próprios guias de uso. Por isso, é essencial consultar a documentação oficial sempre que for utilizar uma API. Nela, você encontrará orientações detalhadas, formas de integração e exemplos de como enviar e receber dados.

Diferenças entre APIs e web services

O conceito de um web service é muito parecido com o de uma API, pois ambos têm o mesmo propósito: permitir a comunicação entre diferentes sistemas. A diferença está na forma como essa comunicação ocorre:

Comparação visual entre web services e APIs: à esquerda, web services com três etapas – sistema interno, envio de requisição XML/JSON/HTTP pela nuvem e cliente; à direita, APIs com trêsz etapas – servidor ou fonte de dados, troca de requisições e respostas da API e dispositivo do usuário.

Web service

Suas requisições são feitas obrigatoriamente por meio de uma rede, como a internet.

API

Nem toda API precisa usar uma rede para funcionar. No entanto, se uma API utiliza a rede para realizar suas requisições, ela é considerada um web service. Em resumo, todo web service é uma API, mas nem toda API é um web service.

Como funcionam os web services

Web services operam por meio de mensagens HTTP, ou hypertext transfer protocol (como GET, POST, PUT e DELETE), utilizando formatos de dados como XML ou JSON. Eles são muito importantes em sistemas modernos, especialmente em aplicativos móveis, nos quais parte das funcionalidades ocorre localmente no dispositivo, mas muitas operações dependem de chamadas na web e são processadas em servidores. É possível citar como exemplos os aplicativos de transporte, as redes sociais e os serviços de delivery.
