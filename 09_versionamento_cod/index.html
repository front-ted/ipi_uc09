Versionamento de código: recuperação de histórico e resolução de conflitos

Sabe-se que os softwares estão em constante evolução, exigindo atualizações frequentes, como novas versões, correções de erros e outras modificações. Para realizar essas manutenções de forma eficiente, é fundamental utilizar ferramentas e métodos específicos de controle. Nesse contexto, os repositórios desempenham um papel essencial, permitindo monitorar e registrar todas as atividades ao longo do ciclo de vida do sistema ou software.

A gerência de configuração de software é uma prática crucial para garantir a rastreabilidade do processo, tornando-se indispensável para equipes de desenvolvimento, especialmente quando múltiplos colaboradores trabalham simultaneamente no mesmo código.

Neste material, você verá como gerenciar o histórico de commits realizados e como solucionar possíveis problemas com as diferentes versões criadas.

Versionamento de código

No contexto do desenvolvimento de software, a organização e o controle das alterações realizadas em um projeto são fundamentais para garantir a eficiência e a colaboração entre os desenvolvedores. Para isso, são utilizadas ferramentas de versionamento, que permitem gerenciar as mudanças no código-fonte de forma estruturada e segura. Essas ferramentas operam por meio de repositórios, que funcionam como bases centralizadas para armazenar e compartilhar o progresso do projeto.

Explicando de uma forma mais clara, um repositório é uma “pasta” onde são disponibilizados os arquivos de um projeto, incluindo-se o versionamento. Um repositório pode ser criado ou baixado com base em alguma hospedagem, de onde são baixados os projetos para uma máquina local, ou, ainda, para onde são enviadas as alterações realizadas. Assim, o desenvolvimento ocorre de forma descentralizada, em que cada desenvolvedor tem uma versão total do repositório em sua máquina local e envia somente os “pedaços” alterados para que todos possam pegar do repositório.

Quando os arquivos são baixados e somente o que foi implementado é enviado, você está trabalhando com versões. Logo, você faz o versionamento do que foi alterado para que se possa manter o controle e a manutenção dos repositórios, evitando que ocorram sobrescritas ou perdas de arquivos.

Ao enviar uma alteração e gerar uma nova versão, é criado um commit, que guardará uma fotografia das últimas versões enviadas ao repositório. No entanto, nesse formato, podem ocorrer situações que gerem conflitos com alterações realizadas por outros desenvolvedores que também realizaram o mesmo processo. Dessa forma, por vezes, uma funcionalidade pode implicar o funcionamento de outra, e isso pode acarretar a geração de conflitos e o mau funcionamento das novas versões do software.

Sendo assim, é importante conhecer os comandos para consultar o histórico de versões anteriores e solucionar possíveis conflitos que possam existir entre elas.

Recuperação de histórico

Existem algumas plataformas de repositórios que podem ser citadas, por exemplo: GitHub, BitBucket e GitLab. O GitHub foi um dos primeiros utilizados em hospedagem de repositórios, e qualquer pessoa que tenha cadastro na plataforma pode colaborar com projetos open source ou privados.

Ao iniciar um novo repositório e realizar atualizações nele, em determinado momento pode ser necessário visualizar o histórico de commits realizados. Essa informação é útil para analisar o que já foi feito e acompanhar, no projeto, quando foi feito e por quem foi feito.

Para isso, basta usar o comando git log em um repositório válido. Esse comando, ao ser aplicado sem argumentos ou filtros, exibe do commit mais recente ao primeiro, trazendo as seguintes informações: SHA-1 do commit, nome e e-mail, data e hora de realização e comentário criado para o commit.

Captura de tela com o comando git log sendo executado.

Essas informações podem ser úteis e tornam-se um recurso interessante quando uma atualização causar um problema no código, pois é possível criar uma rastreabilidade para realizar uma reversão de commit. O comando log pode ser aplicado com argumentos que detalham mais o histórico, como o comando git log -p.

Captura de tela com o comando git log -p sendo executado.

Aplicando o comando, será exibido o que foi realizado de alterações entre um commit e outro, sendo possível aplicar ainda o parâmetro -2, que filtrará para as duas últimas alterações apenas, ficando desta forma: git log -p -2.

O argumento --stat elencará informações resumidas sobre estatísticas do commit, por meio do comando git log --stat, como lista dos arquivos modificados, quantos arquivos foram alterados e total de linhas inseridas e excluídas.

Captura de tela com o comando git log --stat sendo executado.

É possível alterar o formato de exibição das informações, combinando argumentos para realizar exibição em outro formato. Para isso, é possível combinar o comando git log com o argumento --pretty e alguns filtros, como o oneline, ficando, desta forma, git log --pretty=oneline, exibindo as informações de um commit em uma única linha.

Captura de tela com o comando git log --pretty=oneline sendo executado.

O argumento format possibilita que seja gerada uma formatação própria para a exibição das informações, aplicando filtros possíveis de visualizar com o comando git log --format.

Também é possível utilizar o comando --since, que possibilita que uma determinada data seja usada como parâmetro ou informar um período de dias, meses e anos, por exemplo: git log --since=24.weeks, que limitará os commits das últimas quatro semanas.

Ainda é possível filtrar a lista de commits com base em critérios específicos de busca. Por exemplo, a opção --author permite filtrar por um autor específico, enquanto a opção --grep ajuda a localizar commits que contenham determinadas palavras-chave em suas mensagens. Vale lembrar que, caso você queira usar as opções --author e --grep juntas, será necessário adicionar o parâmetro --all-match; caso contrário, o comando retornará resultados que atendam a qualquer uma das condições.

Outra opção de filtro extremamente útil é -S, que aceita um conjunto de caracteres e retorna apenas os commits que introduzem alterações no código no qual esse conjunto foi adicionado ou removido. Por exemplo, se você deseja encontrar o último commit que eliminou ou adicionou uma referência a uma função específica, pode utilizar o seguinte comando: git log -Sfunction_name.

Além disso, uma última opção bastante prática para filtrar resultados no git log é especificar um caminho. Informando um diretório ou o nome de um arquivo, o comando limita os registros aos commits que alteraram especificamente esses arquivos. Normalmente, o caminho é precedido por dois traços (--) para separar os caminhos das demais opções do comando.

Resolução de conflitos

É comum que diferentes desenvolvedores atuem em um mesmo projeto. Sendo assim, para a organização do versionamento de código, é utilizado o recurso de branches, que é um método de criação de ramificações, para facilitar a organização do código, com a finalidade de separar os commits, podendo utilizar uma ou mais ramificações para códigos em desenvolvimento e uma ramificação para o projeto estável.

Contudo, ao possibilitar uma ou mais linhas de desenvolvimento em um repositório e estar trabalhando em um time de desenvolvimento, podem existir discrepâncias entre os commits realizados. Uma situação muito corriqueira de acontecer é a do seguinte exemplo:

João, em sua rotina de trabalho, ao iniciar seu turno, faz um pull para atualizar as últimas alterações realizadas no projeto. Entre outros arquivos, ele faz uma alteração de uma funcionalidade no arquivo cadastroCompetidor do projeto. Antes de fazer uma interrupção para o seu intervalo, ele faz um commit com suas alterações, deixando o versionamento organizado.

No entanto, nesse mesmo projeto, Antônio iniciou sua rotina de trabalho no mesmo momento que João, também realizando um pull e, por uma necessidade, uma alteração no arquivo cadastroCompetidor. Assim como João, antes de fazer uma interrupção para o almoço, Antônio faz um commit com suas alterações. Contudo, ao realizar seu push, este foi rejeitado com uma mensagem de erro: error: failed to push some refs to 'origin/main'.

Essa situação ocorreu porque Antônio tentou enviar seu commit sem as alterações realizadas por João, gerando um conflito entre as versões. Para solucionar essa situação, Antônio precisa sincronizar o repositório remoto com o seu repositório local, fazendo um novo pull.

Ao realizar o novo pull, se não houver conflito entre as versões de arquivos, o repositório é atualizado automaticamente, possibilitando a realização de um novo push. Porém, nesse caso, existe um conflito entre as alterações no arquivo cadastroCompetidor, sendo exibido o arquivo na tela com marcações indicando quais são as alterações feitas localmente e quais são as alterações que estão no repositório remoto, em que:

  • O conteúdo entre <<<<<<< HEAD e======= expressa as alterações locais.
  • O conteúdo entre ======= e >>>>>>> f1a2b3c (ID de um commit) expressa as alterações remotas.

Nesta situação, é necessário escolher uma das três possibilidades:

  • Manter as alterações locais
  • Manter as alterações remotas
  • Combinar ambas as alterações

O indicado é combinar as alterações, realizando um merge entre as versões diferentes, cuidando para que nenhuma alteração seja perdida. Dessa forma, é necessário ter atenção às confirmações que serão realizadas.

Após a solução do conflito, um novo commit é realizado. Como boa prática, recomenda-se adicionar uma mensagem com um comentário explicando a situação e a solução tomada.

Outra boa prática nesse caso seria criar branches para trabalhar de forma separada e, posteriormente, realizar um merge entre as ramificações. Por padrão, ao iniciar um novo repositório, automaticamente é criado uma branch master, sendo a branch padrão do projeto.

No entanto, outro problema que pode ocorrer é realizar um commit defeituoso. Nesse caso, é possível usar o comando git revert, que reverterá as alterações realizadas no último commit, ou outro especificado.

Encerramento

Neste conteúdo, foi possível não apenas acompanhar os comandos essenciais para a consulta do histórico de alterações em um repositório, mas também compreender como as estatísticas geradas pelo Git podem contribuir para uma análise mais detalhada do desenvolvimento ao longo do tempo.

Além disso, foram exploradas diferentes abordagens para a resolução de problemas, utilizando os diversos recursos e comandos que o Git oferece. Esse aprendizado reforça a importância de boas práticas no controle de versões, tornando o processo de desenvolvimento mais organizado, colaborativo e eficiente.
